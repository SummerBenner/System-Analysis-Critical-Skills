1. Briefly summarize the DriverPass project. Who was the client? What type of system did they want you to design?
The DriverPass project was created for Liam, the owner of DriverPass, who wanted a complete online training and scheduling system to help reduce the high failure rate of DMV driving tests. The system needed to support online learning modules, practice exams, and on‑the‑road driving lesson scheduling. It also required multiple user roles—including customers, instructors, the secretary, the IT officer, and the owner—and needed to provide secure access, progress tracking, reporting, and integration with DMV updates.

2. What did you do particularly well?
I did particularly well in identifying, organizing, and documenting system requirements based on the interview transcript. My BRD clearly separated functional and nonfunctional requirements, detailed user roles, and captured the system’s purpose, goals, and constraints.

In the SDD, I successfully included all required UML diagrams—use case, activity, sequence, and class diagrams—and clearly described how each one represented system behavior and structure. I also used the client presentation template correctly and replaced all placeholder text with relevant project information.

4. How did you interpret the user’s needs and implement them into your system design? Why is it so important to consider the user’s needs when designing?
I interpreted the user’s needs by analyzing the interview transcript and translating each requirement into system features, user roles, and technical behaviors. For example, the client emphasized scheduling accuracy, online test availability, and role‑based access, which directly shaped the system’s authentication, scheduling module, reporting features, and online testing components.

Considering user needs is essential because system design must solve the actual problems the client faces. A system that does not reflect user workflows, constraints, or goals will fail to provide value. Incorporating user needs ensures usability, reduces redesign costs, and results in a system that supports real‑world operations effectively.

5. How do you approach designing software? What techniques or strategies would you use in the future to analyze and design a system?
My approach to designing software begins with requirements gathering, followed by translating those requirements into structured models such as use case diagrams, activity diagrams, sequence diagrams, and class diagrams. I rely on both process modeling to understand workflows and object modeling to define system structure.

In the future, I will continue using these techniques while also incorporating:

Iterative refinement of requirements

User‑centered design principles

Clear traceability between requirements and system components

More detailed documentation to support development and maintenance

Improved presentation and communication practices, including complete speaker notes and polished visual materials

These strategies help ensure that the final system is accurate, maintainable, and aligned with user expectations.
